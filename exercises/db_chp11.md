# 11장 연습문제

## 개념퍼즐

**가로**

1. 특정 칼럼을 분리해서 별도의 테이블로 관리하는 것: **수직분할**
4. 논리적 설계 단계의 결과물을 기초로 하여 효율적이고 구현 가능한 물리적 DB 구조를 생성하는 과정: **물리적설계**
7. 릴레이션 스키마와 무결성 제약조건을 기초로 하여 정의한 물리적 데이터 구조, 즉 저장 레코드 양식: **내부스키마**
9. 하나 이상의 테이블에 포함된 데이터의 부분집합으로 구성되는 논리적인 테이블: **뷰**
10. 하나의 업무 단위가 시간의 흐름에 따라 반복적으로 발생하거나 변경되는 이력 또는 진행 이력이 필요한 경우에 추가할 수 있는 테이블: **이력테이블**

**세로**

1. 테이블의 행을 기준으로 분할해서 별도의 테이블로 관리하는 것: **수평분할**
2. 논리적 데이터 구조(논리적 스키마)를 DBMS가 지원하는 효율적이고 구현 가능한 물리적 데이터 구조(내부 스키마)로 변환하는 과정: **물리적스키마모델링**
3. 자주 사용되는 통계 데이터가 여러 테이블에 산재된 데이터를 기초로 하여 생성되는 경우에 추가할 수 있는 테이블: **통계테이블**
5. 트랜잭션 인터페이스 설계를 기초로 하여 구현 단계에서 트랜잭션을 구현하는 데 필요한 세부 사항을 정의하는 과정: **트랜잭션상세설계**
6. 포인터를 사용하여 행의 검색을 촉진할 수 있는 DB 객체: **인덱스**
8. 대용량 테이블의 특정 부분만 주로 사용되는 경우, 해당 부분만 복사해서 추가할 수 있는 테이블: **부분테이블**

## 연습문제

**1. 다음 중 물리적 스키마 모델링의 주요 업무에 대한 설명이 잘못된 것은?** **정답: ②** 내부 스키마를 ER 다이어그램으로 표현한다.
> **해설:** 내부 스키마는 ER 다이어그램이 아닌 DBMS의 특성을 고려하여 '테이블 형태'로 표현합니다. (ER 다이어그램은 개념적 설계 단계에서 사용됩니다.)

**2. 다음 릴레이션 스키마와 무결성 제약조건을 기초로 하여 내부 스키마를 작성하시오. 단, 각 속성의 데이터 타입과 크기는 속성의 특성을 고려해서 임의로 결정하시오.**
> 도서(분류번호, 제목, 저자, 출판사, 발행일, ISBN, 정가)

| No. | 무결성제약조건 | 제약유형 | 관련릴레이션 |
|-----|---------------|---------|-------------|
| 1 | 제목, 저자, ISBN은 널 값을 가질 수 없다. | NOT NULL | 도서 |
| 2 | ISBN은 유일한 값을 가져야 한다. | UNIQUE | 도서 |

**정답 (내부 스키마 작성 예시):**
| 속성 | 칼럼 이름 | 데이터 타입 | 크기 | NULL 허용 | 키 | 기타 |
|---|---|---|---|---|---|---|
| 분류번호 | class_no | CHAR | 10 | | PK | |
| 제목 | title | VARCHAR2 | 100 | N | | |
| 저자 | author | VARCHAR2 | 50 | N | | |
| 출판사 | publisher | VARCHAR2 | 50 | | | |
| 발행일 | pub_date | CHAR | 8 | | | |
| ISBN | isbn | CHAR | 13 | N | | UNIQUE |
| 정가 | price | NUMBER | 7 | | | |

**3. 성능 향상을 위한 물리적 구조 변경 방법 가운데 테이블 추가에 대한 설명이 잘못된 것은?** **정답: ②** 대용량 테이블의 특정 부분만 주로 사용되는 경우 해당 부분만 분리해서 별도의 테이블을 생성할 수 있다.
> **해설:** 대용량 테이블의 특정 부분만 사용할 때, 해당 부분만 '복사'해서 별도의 부분 테이블을 '추가'하는 것입니다. 특정 부분을 '분리'하는 것은 테이블 추가가 아닌 '테이블 분할(수직 분할)'에 대한 설명입니다.

**4. 성능 향상을 위한 물리적 구조 변경 방법 가운데 테이블 분할에 대한 설명이 잘못된 것은?** **정답: ④** 한 테이블 안에서 특별히 빈번하게 갱신되는 칼럼이 있는 경우, 이들 칼럼만 분할해서 관리한다.
> **해설:** 특별히 빈번하게 '갱신'되는 칼럼이 아니라 빈번하게 '검색'되는 칼럼이 있는 경우, 이를 수직 분할하여 물리적 입출력을 줄이고 검색 성능을 높일 수 있습니다.

**5. 성능 향상을 위한 물리적 구조 변경 방법 가운데 칼럼 중복에 대한 설명이 잘못된 것은?** **정답: ①** 자주 검색되는 칼럼이 자주 변경되는 경우 해당 칼럼을 중복시킬 수 있다.
> **해설:** 검색 성능을 높이기 위해 칼럼을 중복시킬 수는 있으나, 데이터 무결성 보장을 위해 중복되는 칼럼은 갱신(변경)이 거의 일어나지 않는 칼럼이어야 합니다. 

**6. 인덱스의 특징이 아닌 것은?** **정답: ③** 기본 테이블을 제거해도 인덱스는 유지된다.
> **해설:** 인덱스는 기본 테이블에 종속된 객체이므로 기본 테이블을 제거하면 해당 테이블의 인덱스도 자동으로 제거됩니다.

**7. 인덱스를 선정할 때의 고려 사항을 잘못 설명한 것은?** **정답: ③** 검색 효율이 가장 중요하므로 가능하면 많은 인덱스를 생성하는 것이 좋다.
> **해설:** 인덱스가 많아지면 검색 속도는 향상되지만 삽입, 삭제, 갱신(DML) 처리 시 오버헤드가 발생하여 전체 성능이 저하될 수 있으므로, 지나치게 많은 인덱스를 생성하는 것은 바람직하지 않습니다.

**8. 다음 제품 릴레이션을 테이블로 표현했을 때, 브랜드명 칼럼과 입고일 칼럼에 각각 인덱스를 생성하는 것이 적절한지 여부를 설명하시오. 단, [참고 사항]을 고려한다.**
> 제품(제품_id, 제품명, 브랜드명, 원가, 판매가, 수량, 입고일, 담당자_id(FK))

**정답 및 해설:**
* **브랜드명:** 인덱스 생성이 **적절하다.** 전체 데이터가 5,000개이고 브랜드 종류가 200개이므로 평균 분포도는 약 0.5% (1/200 * 100)로 매우 좋다. 주로 검색 조건으로 활용되므로 인덱스를 생성하면 검색 효율이 크게 향상된다.
* **입고일:** 인덱스 생성이 **부적절하다.** 데이터의 범위가 10월 7일~28일이고 매주 월요일에만 입고되므로, 실제 값의 종류가 단 4개뿐이다. 분포도는 25% (1/4 * 100)로 분포도가 매우 나쁘며, 10% 미만의 분포도 기준을 충족하지 못한다. 넓은 대상을 검색하므로 전체 테이블 스캔을 하는 것이 오히려 효율적이다.

**9. 음식 테이블에서 ‘음식카테고리’ 칼럼을 기준으로 인덱스를 생성하는 것이 적합한지 여부를 설명하시오. 단, 음식 테이블의 총행 수는 2,000개이고, 음식카테고리는 50개이며, 카테고리별로 음식을 검색하는 일이 빈번하다.** **정답 및 해설:** 인덱스 생성이 **적합하다.** 분포도를 계산해 보면 (1 / 50) * 100 = 2%가 나온다. 평균 분포도가 10% 미만으로 양호하고, 조건(WHERE 절 등)으로 빈번하게 검색되므로 인덱스를 생성하면 검색 효율이 좋아진다.

**10. 다음 중 뷰의 특징이 잘못 설명된 것은?** **정답: ④** 뷰도 물리적인 테이블이므로 저장 공간이 필요하다.
> **해설:** 뷰는 기본 테이블로부터 유도된 논리적인 가상 테이블이므로 시스템에 정의만 저장될 뿐, 데이터를 저장하기 위한 별도의 물리적인 저장 공간이 필요하지 않습니다.

**11. 다음과 같은 뷰 정의서를 기초로 하여 고객등급_뷰를 생성하는 SQL 명령문을 작성하시오.**
> 고객(고객_id, 이름, 휴대폰, 주소, 직장명, 누적구매액, 방문일수, 현재등급)

| 뷰이름 | 용도 | 기본 테이블 | 칼럼 | 데이터 타입 |
|-------|------|------------|------|------------|
| 고객등급_뷰 | 고객의 신규등급판정 프로그램에서사용함 | 고객 | id | VARCHAR2(20) |
| 고객등급_뷰 | 고객의 신규등급판정 프로그램에서사용함 | 고객 | 고객명 | VARCHAR2(20) |
| 고객등급_뷰 | 고객의 신규등급판정 프로그램에서사용함 | 고객 | 구매액 | NUMBER(10) |
| 고객등급_뷰 | 고객의 신규등급판정 프로그램에서사용함 | 고객 | 방문일수 | NUMBER(3) |
| 고객등급_뷰 | 고객의 신규등급판정 프로그램에서사용함 | 고객 | 등급 | CHAR(5) |

**정답:**
```sql
CREATE VIEW 고객등급_뷰(id, 고객명, 구매액, 방문일수, 등급)
AS SELECT 고객_id, 이름, 누적구매액, 방문일수, 현재등급
FROM 고객;
